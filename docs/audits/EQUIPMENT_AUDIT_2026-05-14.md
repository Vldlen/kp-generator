# EQUIPMENT_AUDIT_2026-05-14 — Аудит раздачи оборудования в kp-generator

**Дата:** 2026-05-14
**Метод:** 3 параллельных subagent'а (код-логика / данные каталога / форма и UX) + ручная сверка.
**Скоуп:** как генератор раздаёт оборудование — от формы до строки в КП. Код + данные каталога + бизнес-пробелы модели.
**Цель:** зафиксировать несостыковки перед проектированием новой модели распределения оборудования (классы точек, фискальное оборудование — казначей / казначей+принтер / ФР+ФН).

Подотчёты по веткам — рядом в `docs/audits/`:
- `EQUIPMENT_CODE_AUDIT_2026-05-14.md` (E1–E12)
- `EQUIPMENT_CATALOG_AUDIT_2026-05-14.md` (EC1–EC13)
- `EQUIPMENT_UX_AUDIT_2026-05-14.md` (EU1–EU13)

## Executive summary

Оборудование в генераторе раздаётся **двумя полностью раздельными ветками** (`kiosk` — планшетный комплект, `kiosk_pro` — готовый POS-киоск), и каждая ветка устроена по-своему. Общая картина:

1. **Каталог расщеплён на три несогласованных слоя** — хардкод `catalog.ts`, Google Sheets, Supabase — и разные части кода читают разные слои. Ветка `kiosk` живёт целиком на хардкоде, ветка `kiosk_pro` — на Google Sheets. Кнопка «Синхронизировать каталог» на половину генератора фактически не влияет.

2. **Фискальное оборудование не существует как класс.** ККТ, фискальный регистратор (ФР), фискальный накопитель (ФН), «Казначей», принтеры чеков, сканеры — в модели данных это либо безликая `kiosk_option`, либо вообще не выражается. Для обычного `kiosk` фискалку нельзя добавить ни на форме, ни в калькуляторе. Это главный пробел и причина, по которой задуман новый «класс».

3. **Логика «крепление по типу точки» дублирована в 4 местах тремя разными способами** и уже дрейфует.

4. **Несколько мест, где в КП клиенту молча уходит не то** — fallback на случайный POS-моноблок, бренд вместо обезличенного имени, апрельские хардкод-цены планшетов.

**Найдено: 3 P0, 9 P1, 10 P2, 6 P3.** Плюс — и это главное для следующего шага — **5 структурных бизнес-пробелов модели**, без закрытия которых правила «казначей / казначей+принтер / ФР+ФН» опереться не на что.

Статус: прод работает на «счастливом пути», но модель оборудования негибкая и в ряде мест отдаёт клиенту неверные данные тихо, без сигнала.

## P0 — критично (клиент получает неверное оборудование/цену)

### 🔴 EQ-P0-1. Планшеты для Kiosk берутся из хардкода, а не из каталога

- **Источники:** EU1, EC (раздел планшетов).
- **Файлы:** `page.tsx:406` (выпадающий список из `tablets`), `calculator.ts:78-80` (поиск планшета в `tablets`).
- **Суть:** и форма, и калькулятор для обычного Kiosk используют статический массив `tablets` из `catalog.ts:125` (5 моделей, апрельские цены). Google Sheets на этот блок не влияет вообще. Менеджер физически не может выбрать в форме Kiosk планшет, добавленный в актуальный каталог; в КП всегда уходит хардкод-цена.
- **Почему P0:** цена планшета — самая крупная позиция комплекта (35–65 тыс ₽). Расхождение с актуальным прайсом = неверная смета клиенту. Плюс кнопка «Синхронизировать» создаёт ложное ощущение, что всё свежее.

### 🔴 EQ-P0-2. Kiosk PRO без выбранной модели → в КП молча уходит POScenter Atlas 15"

- **Источники:** E1, E7, EC1, EU7, open question EU.
- **Файлы:** `calculator.ts:170-183` (fallback `posEquipment[0]`), `page.tsx:451` (фильтр моделей по `category === 'kiosk'`), `page.tsx:818` (генерация блокируется только по пустому `client_name`).
- **Суть:** если менеджер не выбрал модель киоска (UI это не блокирует), или каталог недоступен, или выбранный id не нашёлся — калькулятор тихо подставляет `POScenter Atlas 15"` за 39 400 ₽. Усугубляется EC1: встроенный `fallbackCatalog` **вообще не содержит киосков** (моноблоки `catalog.ts` имеют `category:'equipment'` → маппятся в `pos_terminal`, а форма ищет `kiosk`). То есть в режиме fallback список моделей пуст → менеджер не выбирает → fallback на Atlas. Ни тоста, ни пометки.
- **Почему P0:** клиент получает КП на модель киоска, которую менеджер не выбирал.

### 🔴 EQ-P0-3. Анонимизация Kiosk PRO нарушена — в КП уходит бренд

- **Источники:** E2, EC9.
- **Файлы:** `calculator.ts:175` (`name: defaultPOS.name`), `page.tsx:191` (`_kiosk_name = kiosk.name`, не `kiosk.kp_name`).
- **Суть:** для Kiosk PRO в КП попадает `kiosk.name` с брендом («POScenter», «Sam4s», «АТОЛ»), а не обезличенное `kp_name`. У встроенных `posEquipment` поля `kpName` нет вообще. Ветка обычного `kiosk` анонимизирует честно (`kpName || name` везде) — налицо расхождение. Правило воркспейса «в КП без брендов» для PRO нарушается всегда.
- **Почему P0:** прямое нарушение заявленной фичи анонимизации, бренд производителя утекает клиенту.

## P1 — высокий (хрупкость, рассинхрон, ломается при правке каталога)

### 🟠 EQ-P1-1. «Крепление по типу точки» определено в 4 местах тремя способами

E3 + EU9 + EU10 + EC4. Критерий desk/wall/floor живёт в: `mountByType` (хардкод id, `calculator.ts:63-67`), `mountTypeMap` (подстрока в имени, `page.tsx:203-207`), `isNonDefaultMount` (подстрока в group+name, `page.tsx:240-255`), `getDefault()` (почти дословная копия предыдущей, `page.tsx:497-506`). Новая модель киоска с непривычной `group` молча получит `desk` → если она напольная, крепление выпадет из КП без предупреждения. Тип крепления должен быть **полем в каталоге**, не выводиться из имени.

### 🟠 EQ-P1-2. Ветка `kiosk` целиком на статическом каталоге

E4. Кронштейн, адаптер, периферия, крепление пинпада для обычного Kiosk берутся через `getProductById` из встроенного `catalog.ts`, не из Google Sheets. Цены не обновляются синхронизацией; переименование id в каталоге → позиция молча исчезает из КП.

### 🟠 EQ-P1-3. `calculateKP` не самодостаточна

E5. Поля `_kiosk_name/_price/_mount_*/_options_data` заполняются только в `handleGenerate` (`page.tsx:188-232`). Вызов `calculateKP` с «сырой» формой даёт неверный результат (fallback, потерянные опции). Калькулятор не чистая функция от формы — enrichment надо вынести в pure-helper `enrichRequest(req, catalog)`.

### 🟠 EQ-P1-4. Опции теряют id и категорию

E6. `selected_kiosk_options → _kiosk_options_data` маппится в `{name, price}` — id и `category` теряются. В КП все опции получают плоскую `category:'kiosk_option'`. Невозможно отдельно просуммировать или обработать фискальное оборудование.

### 🟠 EQ-P1-5. Категоризация по подстроке имени хрупка

EC3. Подкатегория товара листа «Киоски» определяется подстроками (`крепление`/`принтер`/`сканер`/`ккт`/`фискальн`). «Казначей ФА» (автономная ККТ) не содержит ни одной из них и не начинается с «киоск»/«касса» → уйдёт в дефолтную `equipment`, мимо опций и мимо фискалики. Это тот же класс бага, что недавняя «Касса со сканером».

### 🟠 EQ-P1-6. Поле `group` не нормализуется

EC4. `group` — единственная связь модели киоска с её опциями/креплениями, сравнение строгое (`p.group === kiosk.group`). Парсер делает только `.trim()`, регистр не приводит. «МС 24» ≠ «мс 24» → опции и крепления «исчезают» из формы. Нет валидации, что у группы есть хотя бы одно крепление.

### 🟠 EQ-P1-7. Рассинхрон категорий, нет типизации

EC5. `ProductCategory` объявляет `license`/`service` (не используются), а реально ходящие `kiosk`/`kiosk_mount`/`kiosk_option`/`pos_terminal`/`adapter`/`fiscal` — не объявлены нигде. `DBProduct.category` — нетипизированная `string`, опечатка в Sheets компилятором не ловится. Единого справочника категорий нет.

### 🟠 EQ-P1-8. Для обычного Kiosk нельзя выбрать напольное крепление

EU2. Форма Kiosk даёт только desk/wall. При этом `mountByType` и готовый `kit-floor` напольное поддерживают. Напольный планшетный киоск выразить нельзя.

### 🟠 EQ-P1-9. `parsePrice` обнуляет «цену по запросу»

EC6. Ячейка «по запросу»/«уточняется»/пустая → `sell_price = 0`. Для опций нет фильтра по цене → опция с ценой 0 уходит в КП бесплатной строкой. Легитимная индивидуальная цена неотличима от незаполненной.

## Бизнес-пробелы модели — главное (зачем затеян новый «класс»)

Точные правила комплектования (казначей / казначей+принтер / ФР+ФН) ещё не заданы — будут после этого аудита. Здесь фиксируем **что текущая модель структурно не может выразить**, чтобы новый дизайн строился на полном списке пробелов.

### BG-1. Нет понятия «класс точки»

Комплектация выводится из `license_type` (`kiosk`/`kiosk_pro`) + `kiosk_type` (`desk`/`wall`/`floor`). Нет сущности, которая сказала бы «точка такого профиля требует такой набор оборудования, включая фискальный». Менеджер собирает фискальную часть «на глаз».

### BG-2. Фискальное оборудование не существует как класс

Ни `Product`, ни `DBProduct` не имеют признака типа фискального устройства. ККТ автономная («Казначей»), фискальный регистратор, фискальный накопитель, принтер чеков, сканер ШК — всё либо плоская `kiosk_option`, либо `equipment`/`peripheral`. `CATEGORY_LABELS` объявляет `fiscal`, но ни один парсер эту категорию не выдаёт. **Для обычного `kiosk` фискальное оборудование нельзя добавить вообще** — ни на форме, ни в калькуляторе — хотя реальная inno Kiosk-точка фискальный регистратор имеет.

### BG-3. Нет связи «модель/класс → требуемый фискальник»

Связь оборудования есть только через строковый `group` и только «опции вообще». Нельзя выразить «киоск МС 24 идёт с ФР АТОЛ 30Ф», «к этому классу точки ФР обязателен», «для этого формата достаточно казначея».

### BG-4. Нет понятия комплектности, обязательности, взаимоисключения

Опции — равноправные опциональные чекбоксы. Нет признаков «обязательная позиция», «входит в комплект», «ФР и ФН идут парой», «нельзя выбрать два ФР». Нет валидации «выбран ФР без ФН». `kioskKits` (готовые комплекты) существует, но не используется.

### BG-5. Нет атрибутов фискальных устройств и «цены по запросу»

Срок действия ФН (15/36 мес), версия ФФД, совместимость ККТ↔ФН — полей нет. Признака «цена индивидуальная / по запросу» нет — `parsePrice` сводит нечисловое к 0.

## Рекомендованное направление (вход для проектирования)

Это не готовое решение — это каркас, который нужно наполнить **бизнес-правилами от владельца продукта**:

1. **Ввести явный тип фискального устройства** — enum вроде `fiscal_kind: 'kkt_autonomous' | 'fiscal_registrar' | 'fiscal_drive' | 'receipt_printer' | 'barcode_scanner'`. Поле в каталоге (колонка в Sheets + типизированное поле в `DBProduct`).
2. **Ввести понятие «класс точки» / профиль комплектации** — сущность, которая по выбору менеджера («формат заведения», «способ оплаты», «нужна ли касса») разворачивается в рекомендованный набор: какой фискальник, нужен ли принтер, нужен ли ФН.
3. **Единый справочник категорий** — типизированный enum, один источник истины, отказ от `category: string`.
4. **Тип крепления — поле каталога**, не подстрочная эвристика в 4 местах кода.
5. **Правила комплектности** — обязательность, пары (ФР+ФН), взаимоисключение, валидация комплекта перед выгрузкой.
6. **Каталог — единый слой** (вынос в Supabase с en
um-категорией, либо строгий контракт колонок Sheets), отказ от расщепления «хардкод для kiosk / Sheets для kiosk_pro».
7. **`enrichRequest` как pure-функция** — убрать скрытую зависимость `calculateKP` от `handleGenerate`.

Конкретику пунктов 1–5 (какие именно классы точек, когда казначей vs ФР+ФН, что с чем совместимо) задаёт владелец продукта — это следующий шаг.
