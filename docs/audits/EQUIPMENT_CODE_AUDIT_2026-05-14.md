# Аудит код-логики раздачи оборудования в КП — kp-generator

**Дата:** 2026-05-14
**Скоуп:** только код-логика раздачи оборудования (форма → calculator → equipItems). Аудит каталога и UI/UX — у параллельных агентов.
**Файлы в фокусе:** `src/lib/calculator.ts`, `src/app/page.tsx`, `src/lib/prompt.ts`, `src/lib/catalog.ts`

---

## Summary

Логика раздачи оборудования работает в двух полностью раздельных ветках (`kiosk` и `kiosk_pro`) и в целом корректна для счастливого пути. Но модель **жёсткая и в нескольких местах хрупкая**:

1. Ветка `kiosk` (планшетный комплект) полностью построена на **хардкоженных id товаров** (`mount-onkron-adapter`, `mount-pinpad-bracket`, `mount-onkron-desk-g80`...). Любое переименование/удаление id в каталоге или в Google Sheet тихо выбрасывает позицию из КП — без ошибки, без предупреждения.
2. Понятие «какое крепление для какого типа точки» определено в **4 независимых местах** с **тремя разными способами сопоставления** (id-map, substring-map, substring-эвристика). Они расходятся уже сейчас.
3. Фискальное оборудование (ККТ / ФР / ФН / принтеры / сканеры) **не существует как класс** в модели. Для `kiosk_pro` оно протаскивается как безымянные `kiosk_option` чекбоксы; для обычного `kiosk` его **добавить вообще нельзя**. Нет понятия «класс точки определяет комплектацию».
4. `kiosk_pro` имеет опасный fallback на `posEquipment[0]` — в КП клиенту может уйти не та модель киоска без какого-либо визуального сигнала.

Ниже — находки по приоритетам, далее бизнес-пробелы модели и сильные стороны.

## Findings

### P0 — критично (в КП клиенту может уйти неверная позиция/цена)

#### E1. Fallback `kiosk_pro` на `posEquipment[0]` — в КП уходит произвольная модель
`calculator.ts:170-183`:
```ts
} else {
  // Fallback to old posEquipment[0] if no selected kiosk
  const defaultPOS = posEquipment[0]
  if (defaultPOS) {
    equipItems.push({
      name: defaultPOS.name,          // 'POScenter Atlas 15"'
      category: 'pos_terminal',
      ...
```
Срабатывает когда `req._kiosk_name`/`req._kiosk_price` не заполнены. Это происходит если:
- менеджер не выбрал модель киоска (`selected_kiosk_id === null`) — UI не блокирует генерацию;
- `calculateKP` вызван не из `handleGenerate` (поля `_kiosk_*` заполняются **только** там — см. E5);
- выбранный киоск не нашёлся в `catalog` (id из формы не совпал с актуальным каталогом — `page.tsx:189`).

Результат: в КП уходит **`POScenter Atlas 15"` под именем-с-брендом** (не `kpName`, см. E2), по цене `39 400 ₽`, с `category: 'pos_terminal'`. Клиент получает КП на модель, которую менеджер не выбирал. Нет ни тоста, ни пометки «модель по умолчанию». Это прямой риск отправить клиенту неверное предложение.

**Как проверить:** на форме выбрать `kiosk_pro`, не кликать ни одну карточку киоска, нажать «Рассчитать КП» — в превью появится Atlas 15".

#### E2. Fallback-ветка `kiosk_pro` показывает бренд вместо обезличенного имени
`calculator.ts:175` — `name: defaultPOS.name` (= `'POScenter Atlas 15"'`). В каталоге у `posEquipment` **нет `kpName`** вообще (см. `catalog.ts:48-121` — поля `kpName` отсутствуют). При нормальном пути имя берётся из `_kiosk_name` (= `kiosk.name` из Google Sheets, `page.tsx:191`) — тоже **не** `kp_name`. Правило воркспейса требует анонимизации в КП («планшеты, кронштейны, периферия выводятся без брендов»). Для `kiosk_pro` это правило **нарушается всегда**: и в fallback, и на нормальном пути в КП попадает `kiosk.name`, а не `kiosk.kp_name`. Сравните с веткой `kiosk`, где везде `mount.kpName || mount.name` (`calculator.ts:83,97,...`).

### P1 — высокий (хрупкость / рассинхрон, ломается при изменении каталога)

#### E3. Критерий «крепление ↔ тип точки» определён в 4 местах тремя разными способами
| # | Место | Способ | Критерий desk/wall/floor |
|---|-------|--------|--------------------------|
| 1 | `calculator.ts:63-67` `mountByType` | хардкод id товаров | `desk→mount-onkron-desk-g80`, `wall→mount-onkron-wall-fixed`, `floor→mount-masterhold-kiosk` |
| 2 | `page.tsx:203-207` `mountTypeMap` | substring в названии | `desk→'настольн'`, `wall→'настенн'`, `floor→'напольн'` |
| 3 | `page.tsx:240-255` `isNonDefaultMount` | substring в `group`+`name` | floor если `киоск самообслуживания`/`мс 24`/`мс 32`; wall если `l-240`/`l-320`/`slim`/`настенн`; иначе desk |
| 4 | `page.tsx:497-506` `getDefault()` | substring в `group`+`name` | **дублирует #3** почти дословно |

Проблемы:
- `mountByType` (#1) применяется **только** к ветке `kiosk` — там `floor` маппится на `mount-masterhold-kiosk`, **но форма для `kiosk` даёт выбрать только desk/wall** (`page.tsx:421-443` — две карточки). То есть `kiosk_type === 'floor'` для `kiosk` недостижим через UI, ветка floor в `mountByType` мёртвая для этой ветки.
- #3 и #4 — это **скопированная эвристика**. `getDefault()` (`page.tsx:497-506`) и `isNonDefaultMount()` (`page.tsx:240-255`) проверяют одни и те же подстроки. При добавлении новой модели киоска править нужно **оба** — рассинхрон вопрос времени.
- #2 (`mountTypeMap`) ищет крепление по подстроке в `name` (`page.tsx:211-213`), а #3/#4 определяют дефолт по `group`+`name`. Если в Google Sheet крепление названо иначе (например «Опора напольная» без слова «напольн») — `mountTypeMap` его не найдёт, `_kiosk_mount_*` останутся пустыми, крепление **молча выпадет** из КП (`calculator.ts:186` — `if (req._kiosk_mount_name && ...)`).

**Что при добавлении новой модели:** новый киоск с group, не подпадающей ни под одну подстроку, по умолчанию получит `desk` (`getDefault()` fallback `return 'desk'`, `page.tsx:505`). Если это напольная модель — менеджер увидит дефолт «настольное», а `isNonDefaultMount` сочтёт `desk` дефолтным → крепление не добавится в `_kiosk_mount` → КП уйдёт **без крепления вообще**. Тихо.

#### E4. Хардкод id товаров в ветке `kiosk` — ломается при правке каталога
`calculator.ts` ветка `kiosk` опирается на жёстко зашитые id:
- `calculator.ts:63-67` — `mount-onkron-desk-g80`, `mount-onkron-wall-fixed`, `mount-masterhold-kiosk`
- `calculator.ts:107` — `getProductById('mount-onkron-adapter')`
- `calculator.ts:134` — `getProductById('mount-pinpad-bracket')`

`getProductById` ищет по `allProducts` из `catalog.ts` (`catalog.ts:620-629`) — это **статический встроенный каталог**, не Google Sheets. То есть ветка `kiosk` **полностью игнорирует Google Sheets / Supabase каталог** — планшет берётся из `tablets` (`calculator.ts:79`), кронштейн/адаптер/пинпад из `getProductById` (статика), периферия из `peripherals` (статика, `calculator.ts:122`). Расхождение с веткой `kiosk_pro`, которая целиком на динамическом `catalog`.

Последствия:
- Цены в КП для `kiosk`-комплекта **не обновляются** кнопкой «Синхронизировать каталог» — они зашиты в `catalog.ts`.
- Если id переименуют в `catalog.ts` (`mount-pinpad-bracket` → что-то ещё), `getProductById` вернёт `undefined`, `if (pinpad)` ложь — позиция **молча исчезнет** из КП. Ни ошибки, ни лога.
- Менеджер на форме `kiosk` **не выбирает** кронштейн/адаптер/хаб/пинпад — они зашиты. Подсказка `page.tsx:643-646` обещает «можно заменить любую позицию в превью», т.е. замена делегирована KPPreview, но базовый набор фиксирован кодом.

#### E5. Поля `_kiosk_*` заполняются только в `handleGenerate` — `calculateKP` не самодостаточна
`prompt.ts:25-30` помечает `_kiosk_name/_price/_mount_*/_options_data` как «Internal fields for calculator (populated by page.tsx)». Заполнение — `page.tsx:188-232`, **только** внутри `handleGenerate`. `calculateKP` (`calculator.ts:57`) принимает `ParsedRequest`, но если вызвать её с «сырой» формой (`selected_kiosk_id` заполнен, `_kiosk_*` — нет), для `kiosk_pro` сработает fallback E1 (Atlas 15"), крепление и опции **молча пропадут** (`calculator.ts:186,198` — guard'ы на `_kiosk_*`).

То есть `calculateKP` **не является чистой функцией от формы** — её результат зависит от того, прогнали ли форму через `handleGenerate`. Это скрытая связанность: любой будущий вызов calculator (тесты, server-side, экспорт) обязан вручную повторить enrichment-блок `page.tsx:188-232`. Логику enrichment стоит перенести внутрь `calculateKP` или в отдельный pure-helper `enrichRequest(req, catalog)`.

#### E6. `selected_kiosk_options` → `_kiosk_options_data`: потеря id и категории
`page.tsx:225-230` маппит выбранные опции в `{ name, price }` — **id и `category` теряются**. В `calculator.ts:199-208` все опции получают плоскую `category: 'kiosk_option'`. Значит КП не различает «принтер», «сканер», «ККТ», «ФН» — для шаблона/итогов это однородная масса строк. Невозможно, например, отдельно просуммировать фискальное оборудование или применить к нему особое правило. См. также E9 (бизнес-пробел).

### P2 — средний (edge cases, спорная логика qty)

#### E7. Edge case: `selected_kiosk_id` указывает на отсутствующий в каталоге товар
`page.tsx:189` — `const kiosk = catalog.find(p => p.id === form.selected_kiosk_id)`. Если каталог пересинхронизировали (`GoogleSyncButton`, `page.tsx:811`) **после** выбора модели и id больше нет — `kiosk` будет `undefined`, весь enrichment-блок `if (kiosk)` пропускается, `_kiosk_*` пустые → fallback E1. Менеджер выбрал модель, видел её в карточке, а в КП ушёл Atlas 15". Нет инвалидации `selected_kiosk_id` при ресинке каталога.

#### E8. `devices = 0` и пустые/спорные значения
- `calculator.ts:74,157` — обе ветки оборудования под `req.devices > 0`. При `devices === 0` секция «Оборудование» просто не создаётся — корректно, но для `kiosk`/`kiosk_pro` `update` форсит `Math.max(1, ...)` (`page.tsx:138,144`), `NumberInput` min=1 (`page.tsx:391`). Однако `defaultForm.devices = 1` и при `company === 'bonda'` ставится `0` (`page.tsx:105`). Прямого пути выставить `devices=0` на ИННО-форме нет — edge-case закрыт UI, но не самим `calculateKP`. [unverified — зависит только от UI, при программном вызове calculateKP с devices=0 «Оборудование» молча пропадёт]
- Дубли в `selected_kiosk_options`: `toggleOption` (`page.tsx:596-606`) корректно делает toggle (add/remove), дубль одного id появиться не может. Но **два разных id с одинаковым name+price** (один товар, заведённый в каталоге дважды под разные группы) — оба попадут в `_kiosk_options_data` как две строки. UI дедуплицирует чекбоксы по `name|sell_price` (`page.tsx:587-593`), но дедуп идёт по **отображаемому списку**, а `selected_kiosk_options` хранит конкретный id; если оба id уже в массиве (например после смены модели — хотя `selected_kiosk_id` сбрасывает опции, `page.tsx:151-153`) — задвоение возможно. [unverified — сценарий узкий, проверять через каталог с дублями]

#### E9. qty всех позиций оборудования = `req.devices` — проверка веток
Все позиции в обеих ветках умножаются на `req.devices`:
- `kiosk`: планшет, кронштейн, адаптер, вся периферия, пинпад-крепление — все `qty: req.devices` (`calculator.ts:86,99,112,127,141`).
- `kiosk_pro`: киоск, крепление, каждая опция — все `qty: req.devices` (`calculator.ts:165,191,204`).

По бизнес-правилу («каждый планшет = самостоятельная киоск-станция со своим полным комплектом») для `kiosk` это корректно — комментарий `calculator.ts:69-73` прямо фиксирует, что 14.05 откатили попытку считать хаб/пинпад по `locations`. Для `kiosk_pro` тоже логично — каждый POS-киоск автономен.

**Спорное место — крепление.** `kiosk_pro`: крепление умножается на `devices` (`calculator.ts:191`). Если 5 киосков ставятся в один зал на одну общую стойку/раму — крепление было бы `1`, а не `5`. Сейчас модель не различает «крепление на устройство» vs «крепление на точку». Аналогично для `kiosk` — `mount-masterhold-kiosk` (напольная стойка) по смыслу одна на станцию, тут совпадает с `devices`, но это совпадение, а не выраженное правило. **Нет поля unit-of-scaling у позиции** (`devices` / `locations` / `1`) — масштаб зашит в код per-строку. Это негибко: добавить позицию, которая считается «1 на КП» (например пуско-наладка комплекта), нельзя без правки calculator.

#### E10. `kiosk`-комплект: периферия раздаётся «всё из каталога», без выбора
`calculator.ts:122-131` — цикл `for (const p of peripherals)` добавляет **все 4 позиции** периферии из `catalog.ts:294-339` (БП, кабель, угловой переходник, хаб LAN) в каждый КП. Менеджер не может убрать «угловой переходник» или хаб на форме — только в превью (KPPreview). Если клиенту переходник не нужен — он всё равно в смете до ручной правки. Жёстко, но предсказуемо. Комментарий `calculator.ts:119-121` это осознанно фиксирует («Периферия (дефолт)»).

### P3 — низкий (косметика / тех-долг)

#### E11. Несогласованные `category` у позиций оборудования
В одном КП категории строк: `tablet`, `mount`, `adapter`, `peripheral` (ветка `kiosk`); `kiosk`, `pos_terminal`, `mount`, `kiosk_option` (ветка `kiosk_pro`). Категория `'adapter'` (`calculator.ts:111`) не входит в `ProductCategory` (`catalog.ts:7`) — это свободная строка. Пинпад-крепление получает `category: 'peripheral'` (`calculator.ts:139`), хотя в каталоге это `mount` (`catalog.ts:283`). Категории `LineItem` не типизированы и не консистентны — если шаблон .pptx когда-то начнёт группировать по категории, поведение будет непредсказуемым.

#### E12. `kioskKits` в каталоге не используется логикой раздачи
`catalog.ts:483-532` — массив `kioskKits` (Базовый/Настенный/Напольный/POS) с готовыми составами. `calculateKP` его **не читает** вообще — комплект собирается заново по хардкоду. `kioskKits` — мёртвый код относительно раздачи (возможно используется где-то в UI [unverified — grep по `kioskKits` вне calculator не делался]). Если он задумывался как источник истины для комплекта — он им не стал, и его состав (`kit-wall` включает хаб, `kit-basic` — нет) расходится с тем, что реально раздаёт `calculator.ts` (всегда вся периферия).

## Бизнес-пробелы модели (места жёсткости)

Точные правила фискальной комплектации будут даны позже — здесь только фиксация **где модель негибкая**:

**B1. Нет понятия «класс точки», определяющего комплектацию.** Комплектация выводится из `license_type` (`kiosk` / `kiosk_pro`) + `kiosk_type` (`desk`/`wall`/`floor`). Нет сущности, которая сказала бы «точка типа X требует ФР+ФН, точка типа Y — только принтер чеков». Всё фискальное — ручной выбор менеджера галочками (и то только для `kiosk_pro`).

**B2. Фискальное оборудование не существует как класс.** ККТ / ФР / ФН / принтер чеков / сканер:
- для `kiosk_pro` — приходят как недифференцированные `kiosk_option` (`page.tsx:962-966` присваивает им категорию по подстроке `ккт`/`фискальн`/`принтер`/`сканер`, но в `_kiosk_options_data` категория теряется, см. E6);
- для обычного `kiosk` — **добавить фискальное оборудование вообще нельзя**. Ветка `kiosk` в `calculator.ts:74-154` не имеет ни одной строки про ККТ/ФР/ФН. Форма `kiosk` (`page.tsx:421-647`) не показывает опций фискалки. При этом по заметкам воркспейса у клиента на объекте **есть фискальный регистратор** — то есть реальная inno Kiosk-точка фискальное оборудование имеет, а КП его не выражает.

**B3. Нет правила «когда ФР, когда ККТ, когда принтер».** Модель не кодирует никакой зависимости. Менеджер на `kiosk_pro` ставит галочки вручную, ничто не подсказывает и не валидирует комплект (например «выбран ФР без ФН»).

**B4. Масштаб позиции (`devices`/`locations`/`1`) зашит в код, а не в данные.** Нет поля у позиции/товара, которое говорило бы «эта строка масштабируется по точкам» или «одна на весь КП». Любое новое правило масштабирования = правка `calculator.ts` (см. E9).

**B5. `kiosk_type` перегружен.** Одно поле `'desk'|'wall'|'floor'` означает и «тип крепления планшета» (ветка `kiosk`), и «тип крепления киоска» (ветка `kiosk_pro`). Для `kiosk` значение `floor` недостижимо через UI (E3). Семантика поля разная в двух ветках — расширять рискованно.

## Strengths (сильные стороны)

- **S1.** Ветки `kiosk` и `kiosk_pro` чётко разделены guard'ами `req.license_type === ...` — нет перетекания логики, читается легко.
- **S2.** qty-масштабирование по `devices` для обеих веток обосновано и задокументировано прямо в коде (`calculator.ts:69-73`), включая историю отката неверного толкования аудита от 14.05 — отличная практика «комментарий = ADR».
- **S3.** Каждая вставка позиции защищена `if (product)` — отсутствующий товар не роняет генерацию (хотя и молча выпадает — см. E4).
- **S4.** `_kiosk_options_data` собирается через `.map().filter(...)` с type-guard (`page.tsx:225-230`) — `null` от ненайденных id корректно отфильтровывается.
- **S5.** UI `kiosk_pro` грамотно скрывает селектор крепления, когда альтернатив нет (`totalOptions <= 1`, `page.tsx:515-521`) и автоустанавливает дефолт — менеджер не выберет несуществующее крепление.
- **S6.** Dedup опций по `name|price` (`page.tsx:587-593`) защищает чекбоксы от задвоения при повторе товара в разных группах Sheets.
- **S7.** Анонимизация в ветке `kiosk` сделана честно — везде `kpName || name` (планшет, кронштейн, адаптер, периферия, пинпад).
