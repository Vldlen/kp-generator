# UX / Workflow / Accessibility Audit — kp-generator

**Дата:** 2026-05-14
**Аудит:** менеджерский workflow, preview ↔ pptx parity, точки ошибок, доступность
**Файлы под лупой:** `src/app/page.tsx`, `src/components/KPPreview.tsx`, `src/lib/generatePptx.ts`, `src/lib/calculator.ts`, `src/lib/renderSlideImage.ts`, `src/app/globals.css`, `src/app/layout.tsx`

---

## Summary

Главная находка — **preview в `KPPreview.tsx` и финальный PPTX-слайд из `generatePptx.ts` имеют структурно разные правила раскладки и жёсткие лимиты, которых менеджер не видит**. В preview можно создать секцию «Дополнительно», добавить 4-ю строку в «Услуги» или 8-ю в «Оборудование» — и в .pptx это либо потеряется (строки), либо целая секция не отрендерится. Эталонной финальной картинки до выгрузки нет, менеджер увидит расхождение только когда уже открыл `.pptx` (или, хуже, когда клиент откроет его раньше).

Дальше по списку — отсутствие подтверждений и undo на разрушительные действия (удаление позиции/секции, «Новое КП» сбрасывает всё), потеря работы при случайном закрытии вкладки (нет ни draft в localStorage, ни warning на `beforeunload`), и устаревшая «PDF» терминология в preview, которая создаёт у менеджера ожидание PDF на выходе. Доступность по форме приемлемая (focus rings есть), но клавиатурная навигация по табличному редактору и каталогу-поповеру ограничена, ARIA-атрибутов нет, контраст подписей `text-white/30..40` — ниже WCAG AA.

С точки зрения «мобайл/iPad на встрече» — главная страница верстается под `max-w-3xl`, форма читаемая, но preview-таблица с 6 колонками на iPad-портрете будет скроллиться/жать содержимое.

Сильные стороны: чёткая лицензионно-условная логика формы (qr/ecomm/kiosk/kiosk_pro/findir/bonda_bi), мгновенный пересчёт сумм при редактировании, понятный «обезличиватель» имён (`kpName`), карточная картинка для выбора kiosk PRO, индикатор «загрузки PPTX» (`generating`), fallback на встроенный каталог если Google Sheets лёг.

## Findings

### P0 — большой риск отправить клиенту неверный КП / потерять работу

#### C1. Preview ≠ финальный pptx — жёсткие потолки шаблона невидимы для менеджера
Файл: `src/lib/generatePptx.ts:43-89` (LEFT_CARD: 7 строк, RIGHT_TOP_CARD: 1 строка, RIGHT_BOTTOM_CARD: 2 строки) и `src/lib/generatePptx.ts:172-185`:

```
for (let i = 0; i < card.rows.length; i++) {
  const row = card.rows[i]
  if (i < section.items.length) { ... }
  else { ... } // удаляются ЛИШНИЕ shapes
}
```

Цикл бежит только по `card.rows.length` — если позиций в секции больше, чем строк в шаблоне, **они молча обрезаются**. В preview этих лимитов нет (`KPPreview.tsx:297` — `addItem` без верхней границы). Менеджер видит 9-ю позицию «Оборудование» и сумму с её учётом, но в .pptx строка не появится, итог карточки `subtotal` — тоже из шаблона (не пересчитывается из обрезанного списка), т.е. в .pptx уйдёт сумма с учётом обрезанных позиций — **визуально клиент увидит 7 строк и итог большего размера, чем виден сумма строк**.

Сценарий: менеджер делает крупный комплект из 10 позиций, проверяет сумму на экране, выгружает, отправляет клиенту, клиент звонит «а почему у меня итого 850К, а строк только на 600К?».

#### C2. Секции вне трёх предопределённых заголовков молча выпадают из pptx
Файл: `src/lib/generatePptx.ts:406-412`:

```
const equipSection = kp.sections.find(s => s.title === 'Оборудование') || null
const licSection  = kp.sections.find(s => s.title === 'Лицензии и подписки') || null
const svcSection  = kp.sections.find(s => s.title === 'Услуги') || null
```

В preview `KPPreview.tsx:321-326` (`addSection`) создаётся секция с фиксированным заголовком `'Дополнительно'`. Эта секция в `find` не попадает, **в .pptx не выгрузится вообще**. При этом `grandTotal` на экране (`KPPreview.tsx:251`) считается со всеми секциями, включая «Дополнительно», и попадает в `FOOTER.grandTotal` (`generatePptx.ts:464`). Итог в .pptx — сумма видимых на слайде позиций + невидимая секция → клиент получает несходящийся документ.

Аналогично — если менеджер переименует существующую секцию (заголовок секции в коде не редактируется, но раздел может быть удалён через `removeItem` последней позиции на `KPPreview.tsx:288-290`, и тогда секция исчезает без подтверждения — см. C5).

#### C3. Удаление позиции и секции — без подтверждения и без undo
Файл: `src/components/KPPreview.tsx:467-473`:

```
<button onClick={() => removeItem(si, ii)} ... >
  ✕
</button>
```

Кнопка-крестик с `opacity-0 group-hover:opacity-40` появляется при наведении и удаляет позицию по одному клику. Никакого confirmation. Если в секции остаётся 0 элементов — секция тоже удаляется (`KPPreview.tsx:287-291`), а вернуть её нельзя — кнопки «Отменить» нет нигде.

Сценарий: менеджер вёл КП 10 минут, проматывает курсором по таблице, случайный клик по крестику в строке с большим итогом — позиция исчезла, ещё клик «Назад» в шапке возвращает в форму, но при `handleBack` (page.tsx:246-249) форма сохраняется, а **результат расчёта `kp` пересчитается с нуля при следующем «Рассчитать КП»** — все ручные правки цен/количества/скидок/добавленные строки в preview потеряются. Менеджер бежит к Excel.

#### C4. «Новое КП» сбрасывает всё без подтверждения
Файл: `src/app/page.tsx:251-255`:

```
const handleReset = () => {
  setStep('form'); setForm({ ...defaultForm }); setKP(null)
}
```

И вызов в хедере (`page.tsx:278-280`):

```
<button onClick={handleReset} className="text-sm text-white/30 hover:text-white/60 ...">
  Новое КП
</button>
```

Кнопка слабоконтрастная, рядом с «Редактировать», без confirm. Полминуты работы → один клик → пустая форма. Особенно опасно после генерации pptx — менеджер мог не успеть скачать или попадать в случай «надо ещё одну копию для другого ЛПР» и тут же теряет КП.

#### C5. Edit → Back → Recalculate стирает все правки в preview
Файл: `src/app/page.tsx:246-249`:

```
const handleBack = () => {
  setStep('form')
  // Сохраняем данные формы — можно отредактировать и пересоздать
}
```

И `handleGenerate` (`page.tsx:223-225`):

```
const result = calculateKP(enrichedForm)
setKP(result)
```

Контракт незаметный: «Назад → Рассчитать» полностью пересоздаёт `KPResult` из формы, **отбрасывая всё, что менеджер правил руками в preview**: вручную проставленные скидки, заменённые позиции из каталога, удалённые/добавленные строки, добавленные секции. Опасный сценарий: менеджер открывает preview, видит «не хватает планшета» — идёт в форму поменять количество устройств — нажимает «Рассчитать» — теряет 5 ручных скидок. Никакого предупреждения о потере правок.

#### C6. Нет сохранения draft / нет warning на закрытие вкладки
Глобально — ни `localStorage`, ни `sessionStorage` для формы/КП, ни `beforeunload` listener. При случайном закрытии вкладки/обновлении страницы менеджер теряет всё.

Контекст из CLAUDE.md: «7-10 раз в день», менеджеры работают параллельно с почтой/CRM/Excel — promised lost work.

#### C7. Терминология «PDF» в preview, на выходе .pptx
Файл: `src/components/KPPreview.tsx:513`:

```
<div className="text-sm text-white/50 mb-2">В PDF будет включено:</div>
```

Менеджер видит «В PDF будет включено: 4 слайдов / 2 слайдов», жмёт «Скачать PPTX-презентацию», получает `.pptx`. Это не P0 само по себе, но в сочетании с тем, что клиент может ждать PDF («пришлите КП в PDF»), менеджер может отдать pptx ничего не подозревая, и клиент откроет его в превью Mac/Telegram (где pptx плохо рендерится) — провал.

### P1 — регулярная путаница, замедляет работу

#### C8. Preview-таблица показывает 5 колонок, шаблон pptx считает их иначе
Файл: `src/components/KPPreview.tsx:382-391` (preview-таблица: Наименование / Кол-во / Цена / Скидка / Сумма) и `src/lib/generatePptx.ts:176-180`:

```
xml = replaceShapeText(xml, row.texts[1], String(item.qty))
xml = replaceShapeText(xml, row.texts[2], fmtNum(item.unitPrice))
xml = replaceShapeText(xml, row.texts[3], item.discount > 0 ? `-${item.discount}%` : '—')
xml = replaceShapeText(xml, row.texts[4], fmtNum(item.total))
```

В preview скидка показывается как процент с зелёным цветом (`KPPreview.tsx:183-185`), в pptx — обычным dash или текстом `-N%`. Цена в preview через `formatMoney` («19 400 ₽» — рубли в строке), в pptx через `fmtNum` («19 400» — без валюты). Это не критично, но менеджер может удивиться «а где знак ₽».

#### C9. Кнопка «Назад» — слово «Редактировать», но menager уже редактирует в preview
Файл: `src/app/page.tsx:275-277`:

```
<button onClick={handleBack} className="text-sm text-white/50 hover:text-white transition">
  ← Редактировать
</button>
```

Менеджер на preview уже редактирует (C5), поэтому пункт «Редактировать» читается как «продолжить редактирование» и не намекает на то что он сбросит правки если кликнет «Рассчитать».

#### C10. Selector каталога открывается клику по тексту, кнопка-карандаш редко находится
Файл: `src/components/KPPreview.tsx:216-218`:

```
className="opacity-0 group-hover:opacity-40 hover:!opacity-100 text-white/50 transition p-0.5 flex-shrink-0"
```

Кнопка-карандаш для ручного переименования появляется только при ховере. На iPad/тач — ховера нет, кнопка вообще недостижима. На десктопе её плохо видно: opacity-40 на тёмном фоне — почти невидима.

Сценарий: менеджер хочет переименовать «Планшет Android» в «Планшет Android (модель уточняется)» — кликает на имя, открывается каталог, выбирает то же самое — имя возвращается к шаблонному. Альтернатива (карандаш) спрятана.

#### C11. Hint про клавиатуру в селекторе каталога отсутствует, навигация мышью
Файл: `src/components/KPPreview.tsx:90-96` — `ProductSelector` закрывается только по `mousedown` снаружи. Нет `Escape` для закрытия, нет стрелочной навигации, нет focus trap. Менеджер бесил иногда мышью и не может оставить руки на клавиатуре.

#### C12. NumberInput теряет min при пустом значении
Файл: `src/app/page.tsx:1232-1233`:

```
onBlur={() => {
  setFocused(false)
  const v = parseInt(draft) || min
  onChange(Math.min(max, Math.max(min, v)))
}}
```

`parseInt('') || min` → min. Если менеджер очистил поле «Количество локаций» (нажал backspace всё) и кликнул мимо — значение прыгнет к `min=1`. На локациях ОК, но для `content_items` min=0 — нормально, для `devices` min=1 — менеджер ожидал 0 (хотел временно убрать устройства), получит 1. Мелочь, но накапливается.

#### C13. ID Number-инпута дублируется при «авто-замене»
Файл: `src/app/page.tsx:1211-1213`:

```
if (!focused && draft !== String(value)) {
  setDraft(String(value))
}
```

Вызов `setDraft` прямо в теле компонента (без `useEffect`) — это **anti-pattern**, который React предупреждает в DevTools (warning «Cannot update during an existing state transition»). На проде может работать, но при определённых re-render паттернах словит infinite loop. На UX это влияет тем, что иногда поле дёргается при наборе.

#### C14. После выбора kiosk PRO модели — нет визуального summary «вот что попадёт в КП»
В форме при выборе kiosk PRO появляется список опций чекбоксами (`page.tsx:589-617`), но нет итогового подытога «выбрано N опций на M ₽». Менеджер не видит «где же общая сумма» — кнопка «Рассчитать» работает без summary. На первом расчёте у клиента может оказаться неожиданная цена.

#### C15. Период подписки и количество локаций можно поменять в форме, но цены в preview этого не отражают
Файл: `src/lib/calculator.ts:212-234` — `getLicensePrice` зависит от `qty` (объём), но в preview менеджер не видит «текущая ступенька скидки за объём». Если менеджер увеличит `locations` с 1 → 41, цена за лицензию упадёт (вступает 20% volume discount, см. catalog), но в preview это не визуализируется как «вы получили объёмную скидку».

#### C16. `kpInsertAfter` хардкодом — после слайда 3/4, нет проверки что в шаблоне столько слайдов
Файл: `src/lib/generatePptx.ts:387`:

```
const kpInsertAfter = isQR ? 3 : 4
```

И `page.tsx:521`:

```
sldEntries.splice(kpInsertAfter, 0, kpSldEntry)
```

Если шаблон обновят (кто-то удалит первый слайд в `inno_qr_template.pptx`) — КП-слайд вставится в неправильное место без ошибки. Не UX-bug, но «как менеджер заметит» — никак.

#### C17. ProductSelector показывает альтернативы по `category`, но «Новая позиция» => весь каталог
Файл: `src/components/KPPreview.tsx:53-57`:

```
function getAlternatives(catalog: DBProduct[], productName: string): DBProduct[] {
  const product = findProductByName(catalog, productName)
  if (!product) return catalog // если не нашли — показать весь каталог
  return catalog.filter(p => p.category === product.category)
}
```

Если менеджер только что добавил «Новая позиция» (через `addItem`) — селектор покажет ВЕСЬ каталог сразу (200+ строк, без поиска). Скролл по 200 SKU без фильтра.

#### C18. Нет поиска в каталоге-поповере
Файл: `src/components/KPPreview.tsx:98-133`. Каталог в поповере группируется по категориям, но нет input search/filter. На каталоге 50+ позиций — менеджер мотает 5 секунд чтобы найти OnePlus Pad 3.

#### C19. Loading state каталога не виден пользователю
Файл: `src/app/page.tsx:75-96` — `useEffect` загружает каталог из Google Sheets с fallback. Если Google Sheets медленный (3-5 сек) — менеджер уже зашёл, нажал на kiosk PRO, видит **пустую сетку карточек** (`page.tsx:431-465` — `catalog.filter(...).map`). Никакого skeleton/spinner. Можно подумать «опять не загрузилось» и нажать «Синхронизировать», получив дубль работы.

#### C20. Чекбоксы Kiosk PRO «опции» — две отдельные target-области на одной строке
Файл: `src/app/page.tsx:594-616`:

```
<label className="flex items-center gap-3 cursor-pointer group">
  <div onClick={() => toggleOption(opt.id)} ... > ... checkbox ... </div>
  <div onClick={() => toggleOption(opt.id)} > ... текст ... </div>
</label>
```

`<label>` снаружи, но `<input>` внутри отсутствует — это **не настоящий label**, и клик по label не вызывает `toggleOption` (только нажатие на конкретный `<div onClick>`). Если менеджер случайно кликнет по пустому пространству между чекбоксом и текстом — ничего не произойдёт, менеджер подумает «не работает».

### P2 — улучшения, не критичны

#### C21. Только тёмная тема, нет переключателя
`src/app/globals.css:1-9` — фиксированный `#0f0f1a` фон. У некоторых менеджеров на встречах с клиентами в светлом помещении (солнце на iPad) — тёмные интерфейсы плохо читаются.

#### C22. Контраст подписей `text-white/30` и `text-white/40` ниже WCAG AA
В коде десятки таких подписей: `page.tsx:271, 324, 371, 393, 404` и др. На фоне `#0f0f1a` это `~rgba(255,255,255,0.3)` = ratio ~3.5:1, AA для normal text требует 4.5:1.

#### C23. ARIA-атрибуты отсутствуют
- `RadioCard` — `<button>` без `role="radio"` / `aria-checked` (`page.tsx:830-849`)
- `EditableNumber`, `EditableName` — нет `aria-label` / `aria-live` для обновления суммы
- ProductSelector — нет `role="listbox"` / `role="option"`, `aria-expanded` на trigger
- Screen reader не озвучит изменение grand total после правки

#### C24. Семантическая разметка таблицы — обычный `<table>`, но имя ячейки — `<div onClick>` вместо `<button>`
`KPPreview.tsx:403-421` — ячейка имя — `<div>` с `onClick` без `role="button"`, `tabIndex=0`, без keyboard handler. Клавиатура не работает.

#### C25. Шрифт-размер Inter (default 14-16px) — нормально, но `text-xs` (12px) и `text-[10px]` встречаются часто
`KPPreview.tsx:514, 518` — `text-[10px]` для слайд-инфо, `text-xs` для всех вторичных подписей. Менеджерам старшего возраста — мелко.

#### C26. iPad/мобайл — preview-таблица не оптимизирована
6 колонок (Имя, Кол-во, Цена, Скидка, Сумма, ✕) на iPad-портрете (768px) — будет узко. Никакого `overflow-x-auto` или responsive layout. Менеджер на встрече — мука.

#### C27. Нет автозаполнения «дата КП» в preview
`KPPreview.tsx:360` — `kp.date` всегда `new Date()` из `calculator.ts:329`. Если менеджер делает КП для клиента сегодня но отправляет завтра — дата на КП сегодняшняя. Нет редактирования.

#### C28. Имя файла .pptx — транслитерация русских символов отсутствует
`src/lib/generatePptx.ts:539`:

```
a.download = `KP_${kp.clientName.replace(/[^a-zA-Zа-яА-Я0-9]/g, '_')}_${...}.pptx`
```

Регэксп оставляет кириллицу. Это ОК для macOS, но при отправке через Outlook на корпоративную почту с устаревшим SMTP/MIME может ломаться. Менеджер потом видит у клиента `KP_=?utf-8?...?=.pptx`.

#### C29. Dead code — `src/lib/renderSlideImage.ts` (canvas-рендер) не используется
Grep показывает только определение `renderCommercialSlide` — никто не вызывает. Файл 428 LOC лежит мёртвым грузом. Менеджер этого не видит, но при онбординге нового разработчика — путает (фитбэк-эффект на UX через скорость поправок).

#### C30. AI/OpenAI генерация — нет UI триггера и нет состояния
Несмотря на упоминание OpenAI в CLAUDE.md/KP-SPEC.md, в `page.tsx` и `KPPreview.tsx` — нет ни одного `fetch('/api/openai')` или подобного. Менеджер мог ожидать «текст-помощник для КП» — его нет (либо отключено, либо никогда не было в этом коде). Risk: упоминание в маркетинге → ожидание у менеджера.

## Strengths

1. **Чёткая условная логика формы.** `update(...)` (`page.tsx:98-158`) корректно сбрасывает зависимые поля при смене company / license_type. Менеджер не остаётся с «битой формой».
2. **Обезличивание имён.** `kpName` mapping (`KPPreview.tsx:18-23`) превращает «Onkron G80» в «Кронштейн настольный» автоматически — снижает риск «бренда в КП».
3. **Мгновенный пересчёт сумм.** `recalcItem` + `recalcSection` + `grandTotal` через `reduce` — менеджер видит результат сразу.
4. **Disabled-кнопка «Рассчитать» при пустом client_name** (`page.tsx:799`) — простая, но важная защита.
5. **Loading-state на кнопке «Скачать PPTX»** (`KPPreview.tsx:530-538`) — менеджер видит что идёт работа.
6. **Fallback каталога:** Google Sheets → Supabase → встроенный (`page.tsx:75-96`). Сервис не падает при отказе SaaS.
7. **Карточки моделей kiosk PRO с фото** (`page.tsx:442-465`) — менеджер визуально опознаёт модель.
8. **Адаптивная раскладка КП-слайда в .pptx** (`generatePptx.ts:425-462`) — отсутствующие секции переставляют карточки.
9. **Focus rings на формах** (`focus:ring-1 focus:ring-orange-500/25` и т.п.) — клавиатура хотя бы заметна.
10. **`useCallback` на всех мутаторах** (`KPPreview.tsx:254-326`) — таблица не флапает при печати в любой ячейке.
