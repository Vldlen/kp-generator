# Equipment UX Audit — выбор оборудования менеджером (kp-generator)

Дата: 2026-05-14
Скоуп: форма выбора оборудования (`src/app/page.tsx`) и превью КП (`src/components/KPPreview.tsx`).
Это один из трёх параллельных аудитов equipment-блока. Фокус этого документа — **что менеджер видит и кликает**: форма, превью, расхождения «форма → расчёт → .pptx». Расчётный движок и шаблоны .pptx — в других аудитах.

---

## Summary

Форма читаемая, есть приятные UX-детали (undo-toast, лимиты секций, dropdown с обезличенными именами, confirm на потерю правок). Но в equipment-блоке есть **один структурный P0-баг**, который менеджер не заметит: при выборе планшета для inno Kiosk выпадающий список и расчёт используют **разные источники данных** — менеджер кликает один планшет, в КП попадает другой. Кроме того, для обычного Kiosk менеджер не может выбрать напольное крепление, хотя комплект и калькулятор это поддерживают. Самый крупный бизнес-пробел — менеджер вообще не имеет внятного способа выразить, что точке нужен ККТ / фискальный регистратор / ФН / принтер чеков (для Kiosk — никак, для Kiosk PRO — россыпь безымянных галочек без структуры).

Всего: 1×P0, 4×P1, 5×P2, 3×P3.

## Findings

### P0 — критично

#### EU1 — Выпадающий список планшетов (Kiosk) рассинхронизирован с каталогом расчёта

`page.tsx:406` — список в форме строится из статического импорта `tablets`:

```jsx
{tablets.map(t => (
  <option key={t.id} value={t.id} ...>
    {t.name} — {t.sellPrice.toLocaleString('ru-RU')} ₽ ({t.specs})
```

`tablets` — это хардкод-массив из `catalog.ts:125` (5 моделей: OnePlus Pad 3, Pad Go 2, Honor Pad X8a, Redmi Pad 2 Pro, POCO Pad), **не** живой `catalog` из Google Sheets. При этом весь Kiosk PRO-блок (`page.tsx:450`, `567`) и `ProductSelector` в превью работают с `catalog`, который грузится из Google Sheets (`page.tsx:77-95`).

Последствие: менеджер обновляет каталог из Google Sheets (`GoogleSyncButton`), видит новые/изменённые модели планшетов в превью при замене позиции — **но в форме всё ещё старый хардкод**. Цена в `<option>` (`t.sellPrice`) — тоже из хардкода, может расходиться с реальной ценой из Sheets.

Хуже: `calculateKP` (`calculator.ts:78-80`) ищет планшет тоже в хардкод-`tablets` по `selected_tablet_id`. Так как форма и калькулятор используют один и тот же хардкод-массив, **сам id согласован** — но оба отрезаны от Google Sheets. То есть менеджер физически не может выбрать в Kiosk-форме планшет, добавленный в актуальный каталог; цена в КП — всегда хардкод апрельская. Это прямое противоречие модели «каталог из Google Sheets — единственный источник истины» (комментарий `page.tsx:76`).

Подсказка под списком (`page.tsx:412-416`) тоже читает из `tablets`, не из каталога — усугубляет иллюзию.

**Рекомендация:** строить список планшетов из `catalog.filter(p => p.category === 'tablet')`, и калькулятор тоже переключить на `catalog`. Либо явно задокументировать, что планшеты Kiosk намеренно зафиксированы (тогда убрать «Синхронизировать» как вводящую в заблуждение для этого блока).

### P1 — серьёзно

#### EU2 — Для обычного Kiosk нельзя выбрать напольное крепление

`page.tsx:421-443` — блок «Тип крепления» для Kiosk предлагает **только** desk/wall:

```jsx
<RadioCard ... title="Настольный" .../>
<RadioCard ... title="Настенный" .../>
```

А `floor` отсутствует. При этом:
- `calculator.ts:63-67` `mountByType` содержит `floor: 'mount-masterhold-kiosk'` (стойка напольная) — калькулятор это умеет;
- `catalog.ts:511` `kit-floor` «Напольный киоск» с напольной стойкой существует как готовый комплект.

Если клиенту нужен напольный планшетный киоск (не Kiosk PRO, а планшет на стойке), менеджер **не может это выразить в форме**. Обходной путь — выбрать desk/wall и потом вручную заменить кронштейн в превью через `ProductSelector` — но это неочевидно и теряет напольную стойку из автоподбора. Несимметрично с Kiosk PRO, где floor доступен (`page.tsx:549`).

#### EU3 — `setState` во время рендера в блоке крепления Kiosk PRO

`page.tsx:516-525` — внутри IIFE, исполняемой в JSX рендере, напрямую вызывается `update()`:

```jsx
if (totalOptions <= 1) {
  if (!form.kiosk_type || form.kiosk_type !== defaultMount) {
    update('kiosk_type', defaultMount)   // setState внутри render
  }
  ...
}
...
if (!form.kiosk_type) {
  update('kiosk_type', defaultMount)     // ещё один setState внутри render
}
```

React-анти-паттерн: setState во время рендера вызывает немедленный повторный рендер. В dev-режиме React это терпит (warning в консоли), в проде — лишний цикл рендера. Здесь это **в основном безобидно** (условие быстро сходится, бесконечного цикла нет, потому что `update` сравнивает значение), но хрупко: если `getDefault()` начнёт зависеть от ещё незагруженного каталога, можно получить мерцание. Правильное место — `useEffect` по `selected_kiosk_id`. `[unverified]` — точное поведение в React 18 strict mode не проверял запуском.

#### EU4 — Чекбоксы опций «Дополнительно»: `<label>` без настоящего `<input>`

`page.tsx:614-635` — обёртка `<label>`, но внутри нет `<input type="checkbox">`. Реакция на клик навешена на два отдельных `<div onClick={...}>` (`page.tsx:616`, `629`).

Последствия для UX:
- `<label>` без `for`/вложенного контрола **ничего не делает** — клик по «пустому» месту лейбла (между чекбоксом и текстом, по padding) не сработает. Кликабельны только сам квадратик и блок с текстом.
- Нет нативной клавиатурной доступности: по Tab фокус не попадает на опцию, пробелом не переключить. Для менеджера с клавиатурой — опции недоступны.
- Нет `role="checkbox"`/`aria-checked` — скринридер не озвучит состояние.

Тот же паттерн в чекбоксе «Внедрение и настройка» (`page.tsx:745-764`) — но там одна опция, риск ниже.

#### EU5 — Нет итоговой суммы выбранных опций / нет суммы по комплекту в форме

В блоке «Дополнительно» (`page.tsx:608-639`) каждая опция показывает свою цену, но **нет суммы выбранного**. Менеджер, отметив 4 опции, не видит, на сколько он «разогнал» комплект, пока не нажмёт «Рассчитать КП». Аналогично для Kiosk PRO в целом: модель + крепление + опции — итог появляется только на экране превью. Для Kiosk — комплект из 7+ позиций ценится вслепую. Это провоцирует «прыжки» форма↔превью.

### P2 — заметно

#### EU6 — Загрузка фото киоска из неконтролируемого `image_url`

`page.tsx:463-464`:

```jsx
{k.image_url ? (
  <img src={k.image_url} alt={k.name} className="w-full h-full object-contain p-2" />
```

`image_url` приходит из колонки «Фото» Google Sheets (`page.tsx:1011`) — произвольная строка, которую вписывает кто угодно с доступом к таблице. Нет `onError`-фолбэка: битая ссылка/404 → сломанная иконка картинки вместо placeholder-SVG (а placeholder есть, `page.tsx:466`, но показывается только когда `image_url` пустой, не когда он битый). Менеджер увидит «сломанное» в карточке выбора и может решить, что модель некорректна. Также нет лоадинга/`loading="lazy"` — при 10+ киосках все картинки тянутся сразу.

#### EU7 — Пустое состояние при загрузке каталога Kiosk PRO

`page.tsx:450-477` — карточки киосков рендерятся из `catalog.filter(p => p.category === 'kiosk' ...)`. Пока Google Sheets не загрузился, `catalog` = `fallbackCatalog` (`page.tsx:74`), а в `fallbackCatalog` категория `kiosk` **отсутствует** (fallback строится из `allProducts` = posEquipment+tablets+mounts+peripherals, `catalog.ts:620`; `categoryMap` `page.tsx:39` маппит `equipment→pos_terminal`, киосков там нет). Значит при первом открытии Kiosk PRO менеджер видит **пустой блок без единой карточки и без надписи «загрузка…»** — выглядит как поломка. Когда Sheets догрузится — карточки появятся. Нет ни спиннера, ни текста «Каталог загружается». Аналогично блок «Дополнительно» вернёт `null` (`page.tsx:594`).

#### EU8 — `ProductSelector` позволяет бессмысленную замену, фильтр только по категории, поиска нет

`KPPreview.tsx:65-69` `getAlternatives` фильтрует строго по `product.category`. Это хорошо (не показывает 200 SKU). Но:
- Поиска/фильтра по строке **нет** — при большом каталоге одной категории список прокручивается вручную (`max-h-[50vh]`, `KPPreview.tsx:111`).
- Замена сохраняет `qty`, берёт `unitPrice` из нового продукта (`KPPreview.tsx:320-325`) — но категория строки **не меняется**: можно заменить планшет на другой планшет, это ок; но если строка изначально «свободная» (`item.name === 'Новая позиция'`, `KPPreview.tsx:533`), `getAlternatives` вернёт `[]` (продукт не найден) → `ProductSelector` покажет «Нет альтернатив». То есть для добавленной вручную строки каталог недоступен вообще — менеджер должен вбивать имя и цену руками. Неочевидно.
- Менеджер не может, например, в строке планшета случайно выбрать крепление (категория фильтрует) — это **сильная сторона**, фиксирую в Strengths.

#### EU9 — Расхождение «форма → превью» по креплению Kiosk PRO

`page.tsx:200-219` в `handleGenerate`: крепление ищется в каталоге по подстроке (`mountTypeMap` → `включает 'настольн'/'настенн'/'напольн'`). Передаётся в `_kiosk_mount_name` **только если `isNonDefaultMount`** вернёт true. Логика `isNonDefaultMount` (`page.tsx:240-255`) дублирует эвристику `getDefault()` (`page.tsx:497-506`) — две независимые копии правил «по группе/имени». Если они разойдутся (а они уже почти идентичны, но раздельны), менеджер выберет одно крепление в форме, а в КП попадёт другое (или ничего — дефолт «в комплекте»). Дубль-эвристика без единого источника — точка будущего дрейфа.

#### EU10 — `getDefault()` / `isNonDefaultMount` — хрупкая эвристика по подстроке имени

`page.tsx:497-506` и `240-255` определяют тип крепления по подстрокам в `group`/`name`: `'l-240'`, `'slim'`, `'киоск самообслуживания'`, `'мс 24'`/`'mc 24'` и т.д. Это **захардкоженный список моделей в коде**, тогда как сами модели живут в Google Sheets. Добавят в Sheets новую напольную модель «МС 27» — код её не распознает, дефолт уйдёт в `desk` (`page.tsx:505` `return 'desk'`). Менеджер увидит неверный дефолт крепления и, если не заметит, отправит клиенту настольный киоск вместо напольного. Тип крепления должен быть **полем в каталоге**, а не выводиться парсингом имени.

### P3 — мелочи / полировка

#### EU11 — Подсказка про Kiosk PRO неточна

`page.tsx:648-651`: «Готовый киоск — выберите модель и тип крепления из каталога. **Периферия не требуется.**» Но блок «Дополнительно» (`page.tsx:566`) предлагает принтеры, сканеры, ККТ, ФН — это и есть периферия. Текст «периферия не требуется» противоречит наличию опций периферии прямо под ним. Менеджер может решить, что галочки опциональны/не нужны, и не доставить точке нужный принтер чеков.

#### EU12 — Подсказка про Kiosk перечисляет фиксированный комплект, который не отражает выбор

`page.tsx:644-646`: «Комплект: планшет + кронштейн + адаптер + зарядка + кабель + хаб + крепление пинпада.» Это статический текст. Он корректен относительно `calculator.ts:74-144`, но не упоминает, что **фискального оборудования / ККТ в комплекте Kiosk нет** — а менеджер может предположить, что «крепление пинпада» = эквайринг включён. См. EU13.

#### EU13 — Счётчик «Количество устройств» и «локаций» не связаны визуально

`page.tsx:388-393` «Количество устройств» для Kiosk/Kiosk PRO и `page.tsx:343-345` «Количество локаций» — два независимых числовых поля. Для Kiosk лицензия считается по устройствам, внедрение — по локациям (`calculator.ts:233`, `327`). Менеджер может выставить 5 устройств и забыть про 1 локацию (или наоборот) — никакой подсказки о связи нет. Не баг, но точка ошибки в смете.

## Бизнес-пробелы UX (фискальное оборудование)

Точные правила «какой точке ККТ, какой ФР+ФН, какой принтер» мне не заданы — не придумываю. Фиксирую только то, что **текущий UX не даёт менеджеру внятно выразить фискальную конфигурацию**:

- **Обычный inno Kiosk — способа НЕТ вообще.** Форма Kiosk (`page.tsx:395-443`) предлагает только планшет и тип крепления. Калькулятор (`calculator.ts:74-154`) добавляет планшет, кронштейн, адаптер, периферию, крепление пинпада — **ни ККТ, ни фискальный регистратор, ни ФН, ни принтер чеков**. Если точке Kiosk нужен ФР+ФН — менеджер обязан вручную добавить строки в превью (`addItem`, `KPPreview.tsx:400`) и вбить имя+цену руками, без каталога (см. EU8). Нет ни подсказки, ни чекбокса, ни напоминания, что фискалку надо учесть. Высокий риск выпустить КП на Kiosk без фискального оборудования и не заметить.

- **inno Kiosk PRO — есть галочки, но без структуры.** Блок «Дополнительно» (`page.tsx:566-640`) показывает плоский список опций (`groupOptions` группы киоска + `universalOptions`), где принтеры, сканеры, ККТ, ФН свалены в один список без подзаголовков, без группировки «фискальное / сканирование / печать», без пояснения «когда что ставить». Менеджер видит, например, «ККТ ...» и «ФН ...» как две отдельные галочки и должен **сам знать**, что они идут в паре, что для такой-то точки нужен казначей, а для такой-то — ФР+ФН. UX не направляет: нет «тип точки → рекомендованный набор», нет валидации «выбран ККТ без ФН». Опции категоризируются эвристикой по подстроке имени (`page.tsx:961-966` — `'принтер'`, `'сканер'`, `'ккт'`/`'фискальн'`), то есть даже само попадание опции в список зависит от того, как её назвали в Google Sheets.

- **Нет понятия «тип точки».** Ни в форме Kiosk, ни в Kiosk PRO нет выбора профиля точки (формат заведения / способ оплаты / нужна ли касса), из которого система предложила бы корректный фискальный набор. Менеджер собирает фискальную часть «на глаз». Это главный UX/бизнес-пробел: рекомендую (после получения правил от бизнеса) ввести явный шаг «Фискальное оборудование» с выбором сценария, а не россыпь чекбоксов.

## Strengths (что сделано хорошо)

- **`ProductSelector` фильтрует по категории** (`KPPreview.tsx:65-69`) — нельзя случайно подменить планшет креплением; раньше показывался весь каталог из 200 SKU (комментарий `KPPreview.tsx:62-64`).
- **Undo-toast на удаление позиции/секции** (`KPPreview.tsx:691-721`) с 6-сек таймером и `aria-live` — хорошая защита от случайного клика по «✕».
- **Лимиты секций .pptx видны в превью** (`KPPreview.tsx:501-505` счётчик `N/limit`, `508-518` disabled-кнопка добавления, `checkPptxOverflow` watchdog перед выгрузкой `KPPreview.tsx:441-455`) — менеджер не выпустит КП с обрезанными строками.
- **Confirm на потерю ручных правок** при пересчёте формы (`page.tsx:177-184`) и при «Новое КП» (`page.tsx:264-270`).
- **Обезличивание имён для КП** работает консистентно: dropdown планшетов показывает `kpName` в подсказке (`page.tsx:413-415`), `replaceWithProduct` обезличивает при замене (`KPPreview.tsx:315`, `32-35`).
- **Сброс зависимых полей при смене лицензии/модели** (`page.tsx:119-153`) — при смене `selected_kiosk_id` опции корректно сбрасываются (`page.tsx:151-153`), т.к. зависят от группы.
- **«Крепление в комплекте» без лишнего селектора** (`page.tsx:516-521`) — если у киоска одно крепление, селектор не показывается, просто текст. Меньше шума.
- **Dedup опций по `name+price`** (`page.tsx:587-593`) — если в Sheets опция продублирована под каждую группу, в чекбоксах она одна.

## Open questions

1. **EU1**: планшеты Kiosk намеренно зафиксированы в хардкоде или это недоделка миграции на Google Sheets? От ответа зависит — чинить рассинхрон или убирать иллюзию синхронизации.
2. **EU2**: напольный планшетный Kiosk (не PRO) — это валидный продукт? Если да — нужен `floor` в форме Kiosk. Если нет — почему `kit-floor` есть в каталоге.
3. **Фискалка**: какие реальные правила «тип точки → набор фискального оборудования»? Без них структурный фикс блока «Дополнительно» / добавление шага «Фискальное оборудование» спроектировать нельзя.
4. **Kiosk PRO без выбранной модели**: `calculator.ts:170-182` имеет fallback на `posEquipment[0]` (POScenter Atlas 15"). Менеджер может нажать «Рассчитать» не выбрав киоск — и получить случайный POS-моноблок в КП. Нужна ли блокировка кнопки «Рассчитать КП» при `license_type==='kiosk_pro' && !selected_kiosk_id`? Сейчас блокировка только по пустому `client_name` (`page.tsx:818`).
